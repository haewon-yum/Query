1. Overview & What It Replaces

The geo-recommendation skill replaces a 4-component pipeline (PSO portal UI + n8n orchestration + Google Apps Script + GPT-5) with a single Claude Code skill that handles everything interactively.

Core goal: Given a target app bundle, identify the top 3 geo markets per OS where high-value payers are most likely to be acquired, using a blend of the target app's unattributed user signals and benchmark competitor apps' attributed performance data.

Recommended threshold: spend ≥ $50K D90 → 1,655 candidate bundles. Tight alternative: $100K (1,053 bundles). Both are large enough for genre-level filtering.

⚠️ Gotcha — country coverage: The BQ query showed avg_country_count ≈ 1.0 across all spend buckets when using advertiser.office. This is because advertiser.office is a single value per advertiser, not a geo dimension. For multi-country coverage check, use COUNT(DISTINCT country) from fact_dsp_core where gross_spend_usd > 0.

Auto-suggest logic

6. Scoring Logic

Confirmed from BM_output sheet (weights tab + Weighted Score column):

Spend
0.40
D28 ROAS
0.30
I2P
0.20
ARPPU
0.10

Rationale (from design doc): Spend weighted highest as it most directly reflects investment intensity and real-world performance validation. Weights are intuition-based, not data-derived — a candidate for future calibration.

Scoring steps

  1. Normalize each metric (Spend, D28 ROAS, I2P, ARPPU) to [0,1] within each benchmark title using min-max scaling per OS
  2. Weighted score per title × OS × country: 0.4×norm_spend + 0.3×norm_roas + 0.2×norm_i2p + 0.1×norm_arppu
  3. Average weighted scores across all benchmark titles → one score per OS × country
  4. Rank by avg score within each OS → tag top 3 as 🌟 Top 3
Example output (from BM_output sheet):
iOS NLD: 0.642 🌟 | iOS PHL: 0.635 🌟 | iOS ARG: 0.632 🌟 | iOS ITA: 0.541 | iOS COL: 0.341 | ...

7. LLM Synthesis (Claude)

Claude receives both data blocks and generates the final recommendation narrative, replacing GPT-5.

How it works (confirmed from Apps Script)

The current script uses GPT-5. The skill replaces this with Claude — no external API calls needed, same model running the skill.

10-run frequency aggregation: GPT-5 is prompted to run the geo selection logic 10 times independently, then aggregate results by country appearance frequency across runs. This stabilises recommendations against LLM non-determinism. Claude should do the same — run ranking 10 times with chain-of-thought, count frequency, finalize top 3.

Inputs to Claude synthesis prompt

  • Data A (target unattributed): top 10 countries per OS by unattributed_installs — Country, OS, I2P%, ARPPU, unattributed_installs. Heavier weight.
  • Data B (benchmark scores): OS × Country avg score + priority tag from BM_output
  • Use case context: HVP acquisition for a DSP gaming/consumer advertiser
  • Exclude countries list
  • Which OS values are present in target data (drives iOS/Android output sections)
Key prompt rule (from Apps Script): "Give greater weight to Data A when making your decision." Numbers in Data A must be copied exactly — no approximation or cross-OS mixing. Enforced via structured JSON output schema.

The skill parses this JSON and writes it to the Final_Recommendation sheet tab in plain language — one row per recommended country per OS, formatted for readability by AMs and PSO. An interactive HTML document is generated from the one_pager field in the JSON above and uploaded to the shared Google Drive folder as a .html file. It serves as a client-ready visual summary for AMs — richer than a static slide and shareable via Drive link in the final Slack notification.

Generation approach — Claude-native: Claude already has the full JSON in context and generates the HTML directly — no external API call, no Slides REST complexity. Claude writes polished, styled HTML from the one_pager fields, then the skill uploads the file to the shared Drive folder via the Google Drive API (files.create with mimeType: text/html). Shareable link is extracted from the upload response and sent in Slack.
Shared Drive folder (v1 target): 16wHOr7SDh5_ez_Denqw73opbdQ86loYA — existing shared folder used for geo-recommendation outputs. File naming convention: {bundle_id}_{date}_geo_rec.html

HTML doc structure

  • Header: Bundle identifier, date, use case label
  • Per OS tab/section: Top 3 countries — Country | Strategic Profile | Key Model Strength (I2P, ARPPU, unattributed scale) | Market Intelligence (BM rank + score) | Tactical Tip
  • Signal detail panel: Full top-10 unattributed country table with color-coded I2P and ARPPU bars
  • Frequency chart: 10-run stability — how often each country appeared across runs
  • Alternates: Near-tie countries with profile labels
  • Footer: Analysis window, BM bundle list, methodology note
Scope note — HVP Acquisition only (v1)

The current combining and scoring logic is designed specifically for the HVP acquisition use case (DSP gaming/consumer apps optimizing for payers). For other use cases, the signal set and weighting logic will differ. Example:

  • IAA (In-App Advertising): Only unattributed data is used — no benchmark ROAS scoring (IAA apps typically do not have payer events to anchor BM analysis). Additional signals to include:
    • SDK SOV — the target bundle's share of voice as a publisher on Moloco supply, broken down by OS and geo. A high SOV market indicates strong existing inventory presence and lower incremental acquisition risk.
    • Total Moloco supply spend per OS × geo — indicates supply depth and competition in each market.
    • I2P and ARPPU remain relevant but serve as engagement proxies rather than monetization signals.

Future skill versions should parameterize use case type at the start of the conversation and branch synthesis logic accordingly.

11. Gotchas & Error Handling

Known failure patterns

FailureHow to handle
BQ target query returns 0 rows Do NOT produce a partial recommendation. State that no unattributed install signal was found for the bundle in the given date range. Ask the user to verify the bundle ID, expand the date range, or check if the app is live on Moloco.
BQ BM query returns 0 rows Skip BM step entirely. Proceed to synthesis using Target data only. Note absence of benchmark signal in recommendation output and HTML 1-pager.
No purchase events with revenue > 0 found in pb Flag to user — app may be IAA or event pipeline not set up. Offer to proceed with install/I2P only (no ARPPU). Do not default to a hardcoded event name.
SensorTower returns no genre match for the bundle Ask user to provide genre manually, or fall back to Glean search. Do not auto-select benchmark bundles without genre confirmation.
BM bundle auto-suggest returns < 3 titles Inform the user of the low count. Lower the spend threshold (default $50K D90) and re-run, or allow manual bundle entry.
Slack notification fails Log the failure but do not block the pipeline. Complete remaining steps and surface the Sheets + HTML links directly in the Claude conversation.
Drive HTML upload fails Write the HTML to a local file and offer to retry. Do not block Slack notification — post the Sheets link instead, note HTML unavailable.
All 10 countries are in exclude_countries Surface a warning before synthesis. Ask user to reduce the exclusion list or expand the shortlist beyond top 10.
bundle ID resolves to multiple tracking entities Use resolve_moloco_entity to disambiguate. If still ambiguous, ask user to confirm which advertiser/entity to use.

Do not fabricate

  • Never approximate numbers from Data A in the synthesis prompt — copy verbatim.
  • Never claim a geo is "recommended" if it appeared in fewer than 5 of 10 synthesis runs — surface it as an Alternate only.
  • Never infer genre from app name — always verify via SensorTower or user input.
  • Never hardcode purchase event names — always discover from pb data.

13. Skill File Structure & Reference Folder

The skill folder follows the standard gds-core convention. Reference files contain stable, rarely-changing knowledge that would bloat SKILL.md if inlined — loaded on demand, not pre-loaded every conversation.

Folder layout

gds-core/skills/geo-recommendation/
├── SKILL.md                        ← skill definition (steps, inputs, prerequisites)
├── scripts/
│   ├── purchase_event_discovery.sql   ← Step 0: find revenue events from pb
│   ├── target_query.sql               ← Step 2: full target data pull (pb)
│   ├── bm_bundle_suggest.sql          ← Step 1: D90 spend filter for BM candidates
│   └── bm_query.sql                   ← Step 3: full benchmark data pull (cv + fact_dsp_core)
└── reference/
    ├── scoring_logic.md               ← scoring formula, weight defaults, BM_output structure
    ├── output_schema.md               ← JSON synthesis schema + Final_Recommendation tab layout
    ├── html_template.md               ← HTML 1-pager section spec + field mappings
    └── gotchas.md                     ← full failure pattern list + do-not-fabricate rules

Reference file contents

FileContentsUsed when
reference/scoring_logic.md Min-max normalisation formula per OS group; default weights (Spend 0.4 / D28 ROAS 0.3 / I2P 0.2 / ARPPU 0.1); how BM_output detail rows and country priority summary are structured; how Avg Score is computed across BM titles Step 3 scoring; any question about how scores are derived
reference/output_schema.md Full JSON synthesis schema (all fields, types, constraints); Final_Recommendation tab column spec (Country, Strategic Profile, Key Model Strength, Market Intelligence, Tactical Tip); Alternates format Step 4 synthesis; writing to Final_Recommendation tab
reference/html_template.md HTML 1-pager section-by-section spec: header block, per-OS top-3 table, signal detail panel (I2P/ARPPU bars), 10-run frequency chart, alternates, methodology footnote; field mapping from one_pager JSON → HTML elements; Drive upload procedure Step 5 HTML generation and upload
reference/gotchas.md Full failure pattern table (9 cases from testing); do-not-fabricate rules; known data quirks (tracking_bundle vs app_store_bundle, IFV-first user_id, revenue filter bounds, D28 from cv events not Athena cohort) Any time a query returns unexpected results or synthesis behaviour is unclear
SQL scripts vs reference files: SQL scripts in scripts/ are executed directly by BQ Agent — they are passed as literal SQL, not loaded into Claude's context. Reference files in reference/ are read by Claude on demand when the relevant step is reached or when the user asks a question that requires that knowledge. This keeps context lean for simple runs.
