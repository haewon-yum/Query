Mosaic — Improvement Plan

Roadmap for feedback, discoverability, and Google ecosystem integration
Status: Planning Date: Apr 2026 Questions: 3 areas Current version: v3 (private sub-cats, edit, back-nav)

Mosaic is a live internal HTML report sharing platform for @moloco.com users. Reports are uploaded as self-contained HTML files, organized into categories + sub-categories, with private workspaces and uploader-only edit/delete. The three open questions below explore how to move from a publish-only tool toward a collaborative, discoverable knowledge asset.

💬

Feedback & Discussion on Reports

Area 1
Problem Reports are read-only. There is no way to ask "what does this metric mean?", flag a finding, say "great work", or track action items — readers either send a Slack DM or say nothing.
A Slack post on upload (notify + thread)
When a report is uploaded, auto-post to a designated channel (e.g. #mosaic-reports): "[Title] uploaded by @uploader → link". All discussion lives in the Slack thread. Uses the existing Slack MCP or a simple webhook — no backend storage changes.
Effort: Low (~4h) ✓ Notification-native ✓ Zero infra changes ✗ Discussion disconnected from report ✗ Threads get buried
B In-app comment panel (GCS-backed) Recommended
Comments stored as GCS JSON blobs at meta/comments/{report_id}/{comment_id}.json — same pattern as existing category/report metadata. A collapsible comment panel in the Viewer page shows threaded comments with author avatar, timestamp, and text. Optionally, posting a comment also sends a Slack notification to the report uploader.
Effort: Medium (~2d) ✓ Stays in Mosaic ✓ Private comments possible (same privacy model) ✓ Persistent + searchable ✗ No push notifications unless Slack-wired
Implementation sketch
1
Backend: comment CRUD
GET/POST /api/reports/{id}/comments · DELETE /api/reports/{id}/comments/{cid}
2
GCS schema: meta/comments/{report_id}/{comment_id}.json
{ id, report_id, author_email, author_name, text, created_at, parent_id? }
3
Frontend: CommentPanel in Viewer.tsx
Slide-in panel on right; comment count badge on back-button
4
(Optional) Slack DM to uploader on new comment
POST to Slack webhook / mcp__slack__slack_post_message from backend
C Attach a Google Doc link per report (manual)
Add an optional "Discussion Doc URL" field at upload time. Uploader pastes a pre-created Google Doc link. Mosaic stores and surfaces a button: "Open discussion doc". Zero backend complexity — just a stored URL.
Effort: Trivial (~1h) ✓ No infra ✗ Manual, inconsistent coverage ✗ No structured data
Recommendation: Option A first (Slack post on upload) for immediate zero-effort value, then Option B (in-app comments) as the proper v4 feature. Option C is a stepping stone if Option B is deferred.

Glean Searchability

Area 2
Problem Reports live in GCS behind OAuth. Glean cannot crawl them. Searching "Stonekey CPI analysis" in Glean returns nothing — the knowledge is siloed. We want reports to surface alongside Docs, Slack, and Notion.
A GDrive mirror → Glean indexes automatically Recommended
On each upload, create a companion Google Doc in a shared Team Drive folder (Mosaic Reports/). Content: HTML stripped to plain text + metadata header + link back to the Mosaic viewer. Glean already indexes GDrive (if the connector is active for Moloco) — so the doc appears in Glean search automatically. This also solves Area 3 (comments) in the same pass.
Effort: Medium (~2-3d) ✓ Glean coverage automatic ✓ Also enables GDoc comments ✓ Appears in Drive search ✗ Two sources of truth (GCS + Drive)
Auth blocker Creating docs server-side requires either: (a) a service account with Drive API write access to a shared folder, or (b) using the uploader's Google OAuth token (already acquired at login) with drive.file scope. Current OAuth scope is read-only. Option b requires adding https://www.googleapis.com/auth/drive.file at login — users re-consent once.
B Glean Push API (direct document ingestion)
Glean exposes a REST indexing API (POST /api/index/v1/indexdocument). On each upload, extract text from HTML (BeautifulSoup), then push title + body + viewer URL + metadata to Glean. Glean shows results linking directly to the Mosaic viewer URL.
Effort: Medium (~1-2d) ✓ No Drive dependency ✓ Rich metadata control ✗ Requires Glean API token + datasource config ✗ Viewer URL must be Glean-trusted / accessible ✗ Does not solve comments
Access blocker Need a Glean API token with indexing scope + a custom datasource registered in the Glean admin console. Requires Glean admin access (IT/Eng infra team). Confirm Glean GDrive connector is active first — if it is, Option A is strictly easier.
C Export to Drive on demand (user-triggered)
Add an "Export to Drive" button in the Viewer. When clicked, creates a Google Doc in the user's personal Drive. User-driven, so coverage is inconsistent but requires no service account or scope change (uses existing OAuth token with drive.file).
Effort: Low-Medium (~1d) ✓ No infra setup ✗ Inconsistent coverage (only exported reports searchable) ✗ Doc lives in personal Drive, not shared
Recommendation: Option A (GDrive mirror) — solves Glean and comments together. Confirm whether the Glean → GDrive connector is active for Moloco first; if it is, this is the path of least resistance. Option B (Push API) is a fallback if Drive access is harder to get.

Google Doc Mirror (Searchable + Commentable)

Area 3
Problem HTML files are not natively commentable or searchable in Google Workspace. Connecting each report to a mirrored Google Doc would make it part of the GDrive ecosystem — discoverable in search, commentable by any @moloco.com user, and indexable by Glean.

This is the GDrive mirror approach from Area 2, described in full implementation detail. If Area 2 goes with Option A, this is automatically solved.

End-to-end implementation plan
1
Add drive.file OAuth scope
auth.py: add to SCOPES list → users re-consent on next login
2
Store OAuth access token in session
auth.py: persist token.token + token.refresh_token in session cookie at callback
3
HTML → plain text extraction
services/html_text.py: BeautifulSoup strip_tags() → first 100k chars for Doc body
4
Create Google Doc via Docs API
POST https://docs.googleapis.com/v1/documents with title; then batchUpdate to insert text content + header with link back to Mosaic viewer
5
Share doc with moloco.com domain (anyone can comment)
POST https://www.googleapis.com/drive/v3/files/{id}/permissions — role: commenter, type: domain, domain: moloco.com
6
Store gdoc_id in report metadata
reports.py upload_report: add "gdoc_id": doc_id to data dict → meta.put_report
7
Surface in Viewer + ReportCard
Viewer.tsx: "Open in Docs ↗" button if report.gdoc_id; ReportCard: small Doc icon badge
8
Keep in sync on HTML replace
replace_report_file: re-extract text, batchUpdate doc body; no new doc created

GCS metadata additions

  • gdoc_id — Google Doc file ID
  • gdoc_url — direct viewer link
  • gdoc_synced_at — last sync timestamp

What stays in Mosaic vs. Doc

  • Mosaic: rendered HTML, auth, categories
  • GDoc: plain-text mirror, comments, Glean index, Drive search
Key decision: service account vs. user token If using the uploader's OAuth token, the Doc is owned by the uploader and lives in their Drive. If using a service account, the Doc lives in a shared "Mosaic Reports" Team Drive visible to everyone. The shared folder approach is cleaner for an internal tool but requires Drive API access on the compute service account — check if gds-apac compute SA has this, or request it.

Open Questions & Prerequisites

  • 🔐
    Is the Glean ↔ Google Drive connector active for Moloco?
    If yes, GDrive mirror automatically makes reports Glean-searchable — no Push API work needed. Check Glean admin or ask IT/Eng infra.
  • 🔑
    Does the gds-apac compute SA have Drive API write access?
    Needed for service-account-owned GDoc mirror. Alternative: use uploader's OAuth token (requires adding drive.file scope to login flow).
  • 🏠
    Where should mirrored GDocs live — uploader's Drive or a shared Team Drive?
    Shared Team Drive is cleaner (survives uploader leaving, always visible to team) but requires SA access to that folder. Personal Drive is easier but fragmented.
  • 💬
    In-app comments vs. GDoc comments — which is the primary feedback channel?
    Both can coexist but two comment surfaces is confusing. Preference: GDoc as the canonical comment space if mirror is implemented; in-app panel if staying Mosaic-only.
  • 🔒
    What happens to private sub-category reports in GDrive mirror / Glean?
    A private sub-category should NOT have its GDoc shared with moloco.com — only the owner. Either skip mirror for private reports or share with individual email only.
  • 📣
    Which Slack channel should report notifications go to?
    Options: #mosaic-reports (dedicated), #kr-gds-analytics (team channel), or DM to category subscribers. Needs a channel to be created or designated.
