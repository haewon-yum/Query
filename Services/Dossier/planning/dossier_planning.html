Section 1

Problem Statement

Business context for a key account is scattered across at least three systems — Slack threads, Gong call recordings, and reference docs (Confluence, Notion, Google Drive). No single place gives a GDS or a Growth Manager a coherent picture of what has happened with an account over time.

The consequence is repetitive context-gathering before every meeting or analysis. A GDS preparing for a Netmarble review must manually piece together: what was discussed last week in #ads-netmarble, what commitments were made on the last Gong call, and what the current account strategy doc says. This work is high-effort, low-leverage, and largely invisible.

Specific pain points

  • Pre-meeting prep is manual and fragmented — GMs and GDS each independently re-read Slack history before calls
  • No timeline of key events — it's hard to recall whether a performance issue was raised 2 weeks or 2 months ago, and what the resolution was
  • Meeting recordings are underutilized — Gong calls contain valuable client sentiment and commitments, but are rarely revisited after the fact
  • Onboarding is slow — a new GDS joining the Netmarble account has no structured way to absorb 6 months of account context quickly
  • No account-level memory — decisions made in Slack threads get buried; institutional knowledge leaks out over time
Section 2

Vision & Target Users

Dossier is a Google Cloud Run web service that compiles account business context from Slack, Gong, and reference docs into a single, timeline-first view. It is not a dashboard — it is a living briefing document that grows as the account relationship evolves.

📋
GDS Analyst
Needs operational depth: what was decided, what was escalated, what analysis was run. Uses Dossier to link account events to their own work.
🤝
Growth Manager
Needs rapid context before client calls. Opens Dossier 10 min before a meeting to see recent Slack threads, last Gong call summary, and key open items.
🆕
Anyone Getting Up to Speed
Not just new joiners — anyone who needs to understand the account quickly: a new GDS joining mid-engagement, an APAC GDS manager reviewing the account, or a cross-functional partner preparing for a strategic discussion. Dossier provides the structured narrative that Slack history cannot.

Design principles

  • Timeline-first — every piece of information is anchored to a date; the primary UX is scrolling backward through account history
  • Signal, not noise — summaries over raw transcripts; linked to source for depth
  • Zero manual curation — ingestion is automated; no one should need to "update" Dossier manually
  • Account-scoped — one URL per account; shareable with anyone with a @moloco.com Google account
Section 3

Why "Dossier"

A dossier is a file — a complete, organized record on a subject. The name captures the purpose better than alternatives like "Business Watch" (monitoring/alerting connotation) or "Pulse" (health connotation). Dossier implies depth, history, and intelligence — which is exactly what GDS and GMs need before a strategic conversation.

Naming convention for accounts: Each account gets its own Dossier instance at a stable URL — e.g., /netmarble, /nexon. The service root (/) lists available accounts for the authenticated user.

Feature Specification

5.1 — Timeline View (Core)

The primary view is a vertical chronological timeline. Each event is a card on the timeline, color-coded by source and type. Events can be expanded to show full content, collapsed to show summary only.

📢 EscalationSlackThread summary + permalink + authorRed

🚀 Campaign LaunchSlack / JiraCampaign name, geo, goal, linked ticketGreen
📞 Client CallGongParticipants, AI summary, action items, recording linkBlue
⚠️ Performance AlertSlackMetric flagged, resolution summary, thread linkOrange
📄 Document PublishedGoogle DriveDoc title, author, link, 1-sentence descriptionPurple
🎯 Strategic DecisionSlack / GongDecision text, participants, confidence flagIndigo
🎫 Jira TicketJiraTicket title, status, assignee, linkGray

5.2 — Discussion Digest (Slack)

For each Slack event card, the UI shows:

  • Auto-generated summary (2–3 sentences via Claude API) of the thread content
  • Participants — avatars of who contributed to the thread
  • Slack permalink — deep link to the original thread for full context
  • Tags — auto-classified: escalation / decision / question / fyi

5.3 — Meeting Summaries (Gong)

For each Gong call card:

  • Call metadata — date, duration, Moloco participants, client participants
  • Gong AI summary — pulled directly from Gong's generated summary field
  • Key moments — timestamped excerpts flagged by Gong as significant
  • Action items — extracted from Gong or Claude-parsed from transcript
  • Recording link — direct link to Gong call

5.4 — Reference Docs Panel

A persistent right-side panel (or dedicated tab) listing account reference documents:

  • Account strategy doc (latest version)
  • QBR decks (by quarter)
  • ODSB tickets (open + recently closed)
  • Analysis reports produced by GDS (auto-detected from Drive by account tag)

5.5 — Filters & Search

  • Filter by event type, source, date range, participant
  • Free-text search across all summaries and titles
  • Bookmark / pin key events (stored per-user in browser localStorage for v1)

Section 7

Pilot — Netmarble

Netmarble is the right pilot account: active Slack channel (#ads-netmarble), multiple Gong calls on record, several GDS analyses already completed, and a known account team (GDS + Growth Manager) who can give immediate feedback on usefulness.

Pilot scope

  • Channels: #ads-netmarble (client-specific) + #kor-premium-support-team (KOR GDS × Growth Manager internal discussions) — last 90 days of history
  • Gong: All calls tagged with Netmarble in the last 6 months
  • Docs: GDS analysis reports for 7DS: Origin, Stonekey; account strategy doc if available
  • Jira: ODSB tickets with netmarble label — last 6 months

Pilot success criteria

  • Growth Manager can open Dossier 10 min before a Netmarble call and feel prepared without opening Slack
  • GDS can navigate to the context behind any analysis report without asking teammates
  • Timeline covers at least 90 days of events with no manual curation required
  • Slack summaries are accurate enough that GDS validates them as useful (not misleading)
⚠ Pilot constraint
Gong API access requires Moloco IT to enable API credentials for the service account. Slack bot token requires approval if not already available via MCP setup. Confirm both before starting Phase 1 implementation.

Architecture implications

The signal landscape validates the event taxonomy and surfaces two design decisions:

  • Budget change detection requires a numeric delta signal — keyword rules alone won't catch spend drops. The ingest layer should parse spend numbers from Slack messages and flag changes above a threshold (e.g., >30% drop in budget mentions).
  • Gong call matching by participant domain — Netmarble calls can be matched by participant email domain (netmarble.com attendees) as the primary signal, with title keyword fallback. The April 21 briefing call confirms this approach is sufficient for high-signal calls.
  • Experiment lifecycle is a first-class event type — D28 CPA, ROAS, PA enablement, and RE A/B tests are active simultaneously. Dossier should surface experiment start/end/result as distinct timeline entries, not just doc references.
  • Measurement stack signals are structural, not one-off — Gamesight vs. Appsflyer and iOS PA conflicts are ongoing architectural issues. These warrant a persistent "account context" section (not just a timeline event) to avoid being buried under weekly operational noise.

Section 10

Open Questions

1

Gong API access — is a service account token available?

The prototype will use Haewon's personal Gong MCP token. For Cloud Run deployment, a service account with Gong API access is required. Confirm with IT/Ops whether this can be provisioned, or whether Gong data must be pulled manually and cached.

2

Slack bot token scope — read-only access to #ads-netmarble?

A Slack bot with channels:history and channels:read scopes is required. Confirm whether the existing Moloco Slack workspace has a bot that can be used, or whether a new one needs to be created and approved.

3

Event classification — rules-based or LLM?

Auto-tagging Slack messages as "escalation", "launch", "decision" etc. can be done with keyword rules (fast, cheap) or LLM classification (accurate, slower). For Phase 1 a hybrid makes sense: keyword rules with LLM fallback for ambiguous messages.

4

Gong call → Netmarble matching: how do we identify relevant calls?

Gong calls are not automatically tagged by advertiser. Need a matching strategy: (1) participant email domain matching (netmarble.com attendees), (2) call title keyword matching, or (3) manual tagging in Gong. Confirm which signals are available via the Gong API.

5

Refresh cadence and staleness tolerance?

Nightly refresh is the default plan. For escalation-type events, a 24-hour lag may be too slow. Consider a lightweight webhook trigger from Slack (new message in #ads-netmarble → refresh) for near-real-time updates in Phase 2.

6

What accounts come after Netmarble in Phase 3?

The multi-account design assumes each account has a dedicated Slack channel and Gong tagging convention. Nexon is the natural second account (also KOR GDS). Confirm whether other KOR accounts have sufficient Slack + Gong signal to justify onboarding.
