1. Overview
Why migrate, and what stays the same

Blueprint is currently a Google Apps Script web app — a single Code.js backend and one index.html frontend, deployed inside the Google Workspace environment. Migration to Cloud Run addresses three pain points: execution timeouts (GAS hard-caps at 6 min), caching unreliability (CacheService has a 100KB per-key limit requiring chunked workarounds), and no local dev loop (every change requires a clasp push cycle).

Changing
Backend runtime: GAS → Python FastAPI
BQ client: Advanced Service → google-cloud-bigquery
Cache: CacheService → cachetools.TTLCache
Sheets: SpreadsheetApp → Sheets API v4
Auth: GAS domain lock → Google OAuth 2.0
Frontend API calls: google.script.run → fetch()
Secrets: Script Properties → Cloud Run env vars / Secret Manager
Staying the same
All UI — 100% of HTML, CSS, and rendering JS
All BQ tables and queries (word-for-word)
Google Sheets as the notes store
Cache TTLs (scores 6h, activation 1h)
@moloco.com domain restriction
4 logical API operations (scores, activation, notes GET/POST)
GCP project gds-apac

Frontend diff — 4 call sites changed

Everything else in index.html is untouched. The only edit is swapping the 4 google.script.run call patterns for fetch().

Before — App Script
google.script.run
  .withSuccessHandler(onScoresLoaded)
  .withFailureHandler(function(e){ ... })
  .fetchBlueprintCampaigns();
After — Cloud Run
fetch('/api/scores')
  .then(r => r.ok ? r.json() : Promise.reject(r))
  .then(onScoresLoaded)
  .catch(function(e){ ... });
Before — saveNote
google.script.run
  .withFailureHandler(function(e){ alert(...); })
  .saveNote({ advertiser_id, ... });
After — saveNote
fetch('/api/notes', {
  method: 'POST',
  headers: {'Content-Type':'application/json'},
  body: JSON.stringify({ advertiser_id, ... })
}).catch(function(e){ alert(...); });

app/bq.py

from google.cloud import bigquery

PROJECT_ID = "moloco-ods"
_client = None

def client():
    global _client
    if _client is None:
        _client = bigquery.Client(project=PROJECT_ID)
    return _client

def run_query(sql: str) -> list[dict]:
    rows = client().query(sql).result()
    return [dict(row) for row in rows]

app/cache.py

from cachetools import TTLCache
from threading import Lock

_SCORES_CACHE = TTLCache(maxsize=1, ttl=6 * 3600)
_ACT_CACHE    = TTLCache(maxsize=1, ttl=1 * 3600)
_LOCK         = Lock()

def cached_scores(fn):
    with _LOCK:
        if "v" not in _SCORES_CACHE:
            _SCORES_CACHE["v"] = fn()
        return _SCORES_CACHE["v"]

def cached_activation(fn):
    with _LOCK:
        if "v" not in _ACT_CACHE:
            _ACT_CACHE["v"] = fn()
        return _ACT_CACHE["v"]
About the cache: This is a per-process in-memory cache. It works correctly as long as Cloud Run runs at least 1 instance (--min-instances 1). If it scales to 0 (cold start), the first request after idle will hit BigQuery directly — no data loss, just a ~5–10s wait. We'll set min-instances 1 in deploy.sh to keep the cache warm and avoid cold starts. The tradeoff vs. using Redis/Memorystore: simpler, no extra infra, but each instance has its own cache (if we ever scale to 2+ instances, they'd each fetch BQ independently on first load). For this dashboard's traffic that's fine.

app/notes.py

import os
from googleapiclient.discovery import build
from google.auth import default

HEADERS = ['advertiser_id','advertiser_title','sales_note','gds_note',
           'status','jira_tickets','pso_validated','updated_by','updated_at']

def _sheet():
    creds, _ = default(scopes=["https://www.googleapis.com/auth/spreadsheets"])
    svc = build("sheets", "v4", credentials=creds)
    return svc.spreadsheets(), os.environ["BLUEPRINT_NOTES_SHEET_ID"]

def get_notes() -> dict:
    svc, sheet_id = _sheet()
    result = svc.values().get(spreadsheetId=sheet_id, range="Notes!A:I").execute()
    rows = result.get("values", [])
    if len(rows) <= 1: return {}
    notes = {}
    for row in rows[1:]:
        row += [""] * (9 - len(row))   # pad short rows
        notes[row[0]] = dict(zip(HEADERS, row))
    return notes

Dockerfile — single stage (no frontend build needed)

Unlike Mosaic (React + Python multi-stage build), Blueprint is Python-only — index.html is a static template file copied straight in. Much simpler.

FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt
COPY app/ ./app/
ENV ENV=production
EXPOSE 8080
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8080"]

In order — what to do and in what sequence

Write app/auth.py (copy from Mosaic), app/bq.py, app/cache.py, app/notes.py, app/main.py. Write requirements.txt, Dockerfile, .env.example.
Adapt index.html
Copy index.html from the App Script project. Replace the 4 google.script.run call sites with fetch(). Add an auth gate: if /api/me returns 401, redirect to /auth/login. Save to app/templates/index.html.
Test locally
Set env vars from .env.example (or a local .env). Run uvicorn app.main:app --reload. Verify all 4 API endpoints return data, notes save correctly, OAuth flow completes.
Write setup.sh (one-time GCP setup)
Enable APIs, create Secret Manager secrets, grant the shared SA roles/bigquery.dataViewer on moloco-ods (if not already set) and roles/spreadsheets (or the Sheets API scope). Add redirect URI to the OAuth client.
Write deploy.sh
gcloud builds submit --tag gcr.io/gds-apac/blueprint:latest + gcloud run deploy blueprint with --set-secrets, --min-instances 1, --memory 512Mi. Same pattern as Mosaic's deploy.sh.
First deploy & smoke test
Run deploy.sh. Note the Cloud Run URL. Open it, complete OAuth, verify Scores tab loads data, Activation tab loads, Notes save. Check Cloud Run logs for any BQ or Sheets errors.
Push to moloco/gds
Clone github.com/moloco/gds locally. Copy blueprint-dash/ contents into projects/google_cloud_run/blueprint/. Commit as "feat: add Blueprint Cloud Run service". Push. blueprint-dash/ local dir stays as the primary working copy going forward.
