2. Authentication Flow

User visits any page → auth.require_auth() before-request hook checks session → if no token, redirect to /auth/login
/auth/login generates a CSRF state token, redirects browser to Google's OAuth consent screen with scope: openid email profile bigquery
Google authenticates the user and redirects back to /auth/callback?code=…&state=…
/auth/callback validates CSRF state, exchanges authorization code for access + refresh tokens, fetches user info
Email domain checked — if not @moloco.com → 403 denied; otherwise tokens stored in Flask session
All subsequent BigQuery calls use the user's access token (auto-refreshed if within 5 min of expiry). Queries run as the authenticated user, not as a service account.
/auth/logout clears the session and redirects to /auth/login

4. Code Components

app.py

  • Creates Flask server + Dash app
  • Registers OAuth routes (/auth/login, /auth/callback, /auth/logout)
  • Sets before_request auth guard
  • Session lifetime: 8 hours
  • Entry point for gunicorn (app:server)

auth.py

  • Full Google OAuth 2.0 authorization code flow
  • @moloco.com domain enforcement
  • CSRF state token validation
  • Auto-refresh access token (5 min before expiry)
  • get_current_access_token() used by queries.py

queries.py

  • All BigQuery logic
  • BQ client built from user's OAuth token per request
  • Dropdown loaders: platforms, advertisers, bundles, campaigns, taxonomy
  • load_scores() — main query returning peer + focal rows

layout.py

  • Defines the full Dash UI skeleton
  • Left sidebar: filter panel (Required, Focal Entity, Peer Filters)
  • Main area: custom tab buttons + hidden dcc.Tabs + view toggle
  • All component IDs referenced by callbacks

callbacks.py

  • All Dash reactive callbacks
  • Cascade filter population (platform → advertiser → bundle → campaign)
  • Taxonomy auto-fill on bundle select
  • Run button → BQ query → dcc.Store
  • Tab routing + view rendering (Views 1, 2, 3)

Dockerfile

  • python:3.11-slim base image
  • gunicorn: 2 workers, 120 s timeout
  • Port 8080 (Cloud Run standard)
  • No secrets in image — injected as env vars at deploy time

Taxonomy Join: moloco-dsp-data-view.athena.dim1_app_v3_rmg_vertical

Enriches bundles with vertical / sub_vertical / genre for peer group construction.

Query Output Schema (load_scores())

Returns a single DataFrame with two row types, distinguished by is_focal:

is_focalDescriptionUsed for
False Global peer pool — all platforms, one row per bundle × pillar × blueprint_index. Bundle-level spend-weighted scores. Box plot distribution, peer benchmark stats (median, IQR, percentile)
True Focal entity rows — platform + entity filtered, campaign-level grain. Focal diamond on chart, detail table, recommendations, campaign range

Peer Group Logic

LevelConditionFallback
Genre≥ 10 bundles at this genreTry sub-vertical
Sub-vertical≥ 10 bundles at this sub-verticalTry vertical
Vertical≥ 10 bundles at this vertical"insufficient" — banner warning shown

6. Filter Logic

Filter typeApplied toFilters
Taxonomy filters Global peer pool only (global_enriched) Vertical, Sub-vertical, Genre — via dim1_app JOIN
Context filters Both peer pool and focal entity OS — applied as UPPER(b.os) = @os (case-insensitive)
Focal filters Focal entity only (focal_enriched) Advertiser ID, App Bundle, Campaign ID
Spend filter Peer pool only spend_L7 > 0 — ensures only active bundles in peer distribution

Taxonomy filters are intentionally NOT applied to focal_enriched — a focal bundle with a NULL dim1_app taxonomy (new/untagged) would be silently excluded if taxonomy filters were applied there.

7. UI Views

Tab 1 — Overall & Pillar Scores

  • Horizontal box plots — one per pillar + overall
  • Peer distribution (gray boxes) + focal diamond (blue = bundle, red = campaigns)
  • Headline callout: percentile, most lagged / most advanced pillar
  • Summary table: Bundle Score | Peer Median | Peer IQR | Gap | Status
  • Campaign view adds "Campaign Range" column (min–max across campaigns)
  • Toggle: Bundle view vs Campaign view

Tab 2 — Pillar Drill-Down

  • Select pillar → sub-pillar box plots within that pillar
  • Callout: pillar percentile + most lagged / strongest sub-pillar
  • Peer Benchmark table: per-sub-pillar gap vs peer median
  • Sub-pillar dropdown filters Campaign Detail table
  • Campaign Detail: Campaign | Spend L7 | Score | Detail | Recommendation

Tab 3 — Peer Leaderboard

  • Ranked table of all peer bundles by overall Blueprint score
  • Focal bundle highlighted in blue (★) with blue background row
  • Columns: Rank | App | Bundle | Overall | per-pillar scores
  • Score cells color-coded: green ≥ 80 / orange ≥ 50 / red < 50
  • Focal rank shown in subtitle (e.g. "focal bundle ranked #12")

9. Known Constraints & Gotchas

IssueDetail
BQ job permission required Users must be in gcp-bigquery-user or gcp-gds-apac Google Groups to run queries. Auth succeeds but dropdowns appear empty without this.
OS field mixed case Blueprint table stores os in mixed case (e.g. "Android", "android"). Always filter with UPPER(b.os) = @os.
Taxonomy filters on focal entity Never apply vertical/sub_vertical/genre filters to focal_enriched — untagged bundles have NULL dim1_app joins and would be silently excluded.
spend_L7 in UNION Campaign-level spend_L7 only exists in focal rows. Peer rows carry CAST(NULL AS FLOAT64) AS spend_L7 to keep the UNION schema aligned.
gunicorn session affinity 2 workers share the same Flask secret key so sessions are valid across workers. Scaling to multiple instances would require a shared session store.
Cold start latency First BQ query after a cold start can take 10–20 s. The "Querying BigQuery…" spinner covers this.
