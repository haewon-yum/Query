Key Challenge: Auth Is Fundamentally Different

Critical difference: Seller Insight uses user-delegated BQ credentials — OAuth requests the bigquery scope and executes queries as the logged-in user. This is not an accident: the SA used by Blueprint-dash does not have access to ads-bpd-guard-china. Seller Insight was built specifically to bypass this SA permission gap.

Any integration strategy must preserve user-level BQ execution for Seller Insight queries, or separately resolve the SA permission issue on ads-bpd-guard-china.

Integration Options

Option A

iframe Embed — Separate Services

Add a "Seller Insight" tab to Blueprint-dash's index.html that renders the existing Seller Insight Cloud Run URL inside an <iframe>. Both services continue running independently.

Pros
  • Zero code change to Seller Insight
  • 1–2 hour implementation
  • Independent deployments
  • No framework conflict
Cons
  • User logs in twice (separate cookies)
  • iframe UX friction (double scrollbar, sizing)
  • Two separate Cloud Run services to maintain
  • No shared state (can't pass filters from Scores tab)

Effort: ~2h · Auth: double login · Maintenance: 2 services

Option B — Recommended

Mount Dash as WSGI Sub-App Inside Blueprint-dash

Mount the Seller Insight Dash/Flask app at /seller-insight inside Blueprint-dash using WSGIMiddleware. Blueprint-dash's FastAPI handles /, /api/*, and /auth/*; Seller Insight handles /seller-insight/*. Both share the same OAuth callback but maintain separate session storage.

FastAPI (port 8080)
├── /                   → Blueprint-dash (scores, activation)
├── /api/*              → Blueprint-dash BQ endpoints (SA-level)
├── /auth/*             → Unified OAuth (requests bigquery scope)
└── /seller-insight/*   → Dash app (WSGIMiddleware, user-level BQ)
Auth flow: Upgrade Blueprint-dash's OAuth to request the bigquery scope. On login, store the user's access token in an httpOnly cookie or extend the existing signed session. Seller Insight reads the token from the shared session to instantiate its BQ client. One login covers both.
Pros
  • Single login, single URL
  • Single Cloud Run deployment
  • No rewrite of Seller Insight callbacks/charts
  • User-level BQ preserved (no SA permission needed)
Cons
  • ASGI/WSGI bridging requires care (path stripping)
  • Auth refactor: Blueprint-dash must request bigquery scope
  • Dash's /_dash-* internal routes need allow-listing in require_auth
  • Seller Insight's server.before_request hook conflicts with FastAPI auth — must be removed

Effort: ~2 days · Auth: single login · Maintenance: 1 service

Option C

Full Port — Seller Insight to FastAPI + Plotly.js

Port Seller Insight's Dash callbacks to FastAPI endpoints. Replace Plotly Dash charts with Plotly.js in the vanilla JS frontend. Eliminates the Dash dependency entirely.

Pros
  • Single framework, cleanest architecture
  • Full control over UI/UX consistency
  • No WSGI/ASGI bridging
Cons
  • ~1–2 week effort to port all callbacks + charts
  • Box plot + leaderboard logic is non-trivial in vanilla JS
  • High regression risk on complex peer group logic

Effort: ~1–2 weeks · Auth: single login · Maintenance: 1 service

Recommendation: Phased Approach (A → B)

Start with Option A (iframe) to unblock users immediately. Migrate to Option B (WSGI mount) to fix the double-login UX and consolidate to a single service. Option C is a future consideration once the integration is stable.

Phase 2 — Unified Auth + WSGI Mount ~2 days

1
Upgrade Blueprint-dash OAuth to request bigquery scope

In app/auth.py, add https://www.googleapis.com/auth/bigquery to the OAuth scope. Store the returned access_token and refresh_token in the existing itsdangerous session alongside email and name. Add a token-refresh helper.

2
Strip Seller Insight's standalone auth

Remove server.before_request(auth.require_auth) from app.py. Remove the Flask /auth/* routes — Blueprint-dash's /auth/* now handles login for both. Seller Insight's queries.py reads the token from the shared cookie/session.

3
Mount Dash app via WSGIMiddleware 
from starlette.middleware.wsgi import WSGIMiddleware
from seller_insight.app import server as dash_server

app.mount("/seller-insight", WSGIMiddleware(dash_server))

Move the seller_insight/ directory into the Blueprint-dash repo. Update all internal Dash asset paths to be prefixed with /seller-insight (set requests_pathname_prefix="/seller-insight/" and routes_pathname_prefix="/seller-insight/" in the Dash constructor).

4
Update token injection in queries.py

Replace Seller Insight's get_current_access_token() (which reads from Flask session) with a version that reads from Blueprint-dash's shared session cookie.

5
Update requirements.txt and Dockerfile

Add Dash and Plotly dependencies to Blueprint-dash's requirements.txt. Verify memory limit (512 MiB → may need 1 GiB due to Plotly + pandas at query time).

6
Replace iframe with direct tab navigation

Update index.html to navigate to /seller-insight instead of rendering an iframe. Or embed as a true in-page section by keeping the iframe but pointing it at the local /seller-insight path (same domain = no second login).

Phase 3 — Full Port (Future) ~1–2 weeks

Once Phase 2 is stable, evaluate porting Seller Insight's Dash callbacks + Plotly charts to vanilla Plotly.js + FastAPI endpoints. The box-plot peer distribution and leaderboard ranking logic are the most complex parts to port. Treat this as a separate project with its own plan doc.

Open Questions

  • SA permission on ads-bpd-guard-china: If the SA is ever granted access to this project, Seller Insight could drop user-level BQ execution and use the SA like Blueprint-dash. This would simplify Phase 2 significantly. Worth tracking.
  • Memory limit: Dash + Plotly adds substantial runtime memory (Plotly chart objects, pandas DataFrames per live query). Current Cloud Run limit is 512 MiB — likely needs 1 GiB minimum for Phase 2.
  • Seller Insight's Dash app path in the repo: Does seller_insight live as a sibling service in moloco/gds or is it merged into the Blueprint blueprint path? Decide before Phase 2 to avoid repo restructure mid-implementation.
  • Cookie size: Adding access_token + refresh_token to the itsdangerous signed cookie increases it by ~500–800 bytes. Signed cookies are fine up to ~4 KB; should be within limit but worth verifying.
  • Session secret sharing: Phase 2 requires that the token stored by Blueprint-dash's OAuth is readable by the Dash app's query layer. Confirm whether itsdangerous cookie or a server-side KV store is the right carrier.
