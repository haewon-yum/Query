GDS AI Development Framework

A clear guideline for the GDS team on leveraging AI — accelerating speed of development, enabling knowledge sharing, and upleveling team AI proficiency without slowing down exploration or new invention.

01 Purpose & Scope

Why this framework exists and what it governs.

⚡ Accelerate

Reduce time from question → insight → deployed service by standardizing how AI tools are used across the team.

🔁 Share

Prevent knowledge fragmentation. When one team member solves a hard problem, everyone benefits — immediately and permanently.

📈 Uplevel

Build a clear proficiency ladder so every GDS member can grow from user → builder → architect of AI-powered analytics.

Core tension to manage: Standardization should accelerate exploration, not block it. The framework defines on-ramps (how work gets promoted from personal → team → production), not gates that slow individual work.

03 Baseline AI Environment

One-stop setup every GDS team member runs on day one. Managed in GitHub — updates propagate to everyone via git pull.

Key insight: Domain knowledge (CLAUDE.md, memory, validated SQL) must be team-owned and version-controlled in GitHub — not scattered in personal ~/.claude/ directories. Searchlight and BQ Agent are only as good as the context they're grounded in.

Proposed repo: moloco/gds-ai-toolkit

gds-ai-toolkit/ ├── CLAUDE.md # Team-shared base context (GDS conventions, anti-patterns) ├── .mcp.json # Standard MCP connections (BQ Agent, Glean, Jira, Slack, Jupyter, Drive) ├── memory/ # Shared memory templates — team pulls, personalizes on top │ ├── bq_tables.md # Verified BQ schemas + gotchas (app_market_bundle nesting, etc.) │ ├── client_profiles.md # Key clients: bundle IDs, KPI targets, contacts │ └── conventions.md # GDS SQL patterns, naming, notebook structure ├── sql/ # Validated, reusable SQL patterns │ ├── market_share_soi.sql │ ├── campaign_performance.sql │ └── launch_analysis_template.sql ├── skills/ # GDS-specific Claude Code skills (pre-installed) ├── setup.sh # One-command setup: installs Claude Code, links MCP, populates memory └── README.md

05 Lifecycle Tiers

How work moves from personal exploration to team asset to production service. The tier defines the required standard — not a gate, but an on-ramp.

Tier 1
Explore
  • One-off query, notebook, or local skill
  • No review required
  • No documentation needed
  • Owned by individual
  • Not shared by default
Tier 2
Share
  • Posted to GDS shared folder or Slack
  • Basic explanation of what/why
  • SQL must run without errors
  • Peer awareness (no formal review)
  • Colab or HTML format preferred
Tier 3
Standardize
  • Speedboat skill or gds-ai-toolkit SQL
  • Peer review required
  • Input/output documented
  • Owner assigned
  • Validated against known baseline
Tier 4
Productionize
  • GCR service with public URL
  • Deployment checklist passed
  • SLA & monitoring defined
  • On-call / owner registered
  • Deprecation policy set

Promotion criteria

From → ToTriggerMinimum bar
T1 → T2 Another team member would use this SQL runs; brief context written in Slack or Colab header
T2 → T3 Used by >1 person or repeatedly requested Peer review, owner named, documented inputs/outputs
T3 → T4 Business team needs to access it without GDS involvement GCR deploy checklist, monitoring, SLA commitment

  • Client profiles (Notion)
  • Moloco wiki (Notion/Confluence)
  • BQ schemas (live via BQ Agent)

🔍 Index & Retrieval

  • Glean indexes Notion + Drive
  • GitHub code search
  • BQ Agent for live schema
  • CLAUDE.md loaded at session start
→

🤖 Grounded Agents

  • Searchlight (/investigate)
  • BQ Agent (SQL execution)
  • MOBIUS (domain-aware queries)
  • Future: /domain-context skill
Searchlight as retrieval engine, not knowledge store. Searchlight surfaces knowledge from Glean, Notion, and BQ — but the knowledge must first be written somewhere Glean can index. The missing piece is a contribution workflow: where does a team member's discovered insight land so everyone benefits?

07 Agent Architecture

How existing and future agents are composed. Goal: a "Virtual GDS" that any stakeholder can interact with.

BQ Agent

Responsibility: SQL execution, schema discovery, query validation.
Interface: Invoked by MOBIUS / Searchlight, never directly.

MOBIUS

Responsibility: Domain-aware query planning. Grounds BQ queries in Moloco context before execution.
Interface: Claude Code chat.

Searchlight

Responsibility: Cross-domain RCA — BQ + Glean + Jira + Drive + Looker in one investigation.
Interface: Skill-prefixed CLI (/investigate, /campaign-status).

Future: Virtual GDS Agent

A composed agent that business stakeholders (Sales, AM) can query directly — answering campaign questions, surfacing past analyses, and routing to deeper investigation when needed.

Not yet defined. Architecture decisions needed: orchestration pattern (sequential vs. parallel agents), entry point (Slack bot / web UI / Claude Code), scope of access, and handoff protocol to human GDS analyst.

08 AI Proficiency Ladder

How GDS members grow from AI users to AI architects. Each level has clear capabilities and contributions.

Level 1
User
  • Uses Claude Code for daily analysis
  • Runs existing skills (/investigate, /campaign)
  • Follows GDS CLAUDE.md conventions
  • Can read and run shared SQL from toolkit
Level 2
Contributor
  • Contributes to gds-ai-toolkit (PR)
  • Writes validated SQL → submits to shared library
  • Builds and shares Tier 2 analyses
  • Identifies when to use Haiku vs. Sonnet vs. Opus
Level 3
Builder
  • Builds Tier 3 Speedboat skills
  • Designs multi-step agent workflows
  • Deploys Tier 4 GCR services
  • Writes CLAUDE.md conventions for others
Level 4
Architect
  • Designs GDS agent architecture
  • Owns gds-ai-toolkit direction
  • Defines skill validation standards
  • Mentors others across all levels
